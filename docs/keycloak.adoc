= Keycloak — Guía para principiantes
:toc: left
:toclevels: 3
:source-highlighter: highlight.js

== ¿Qué es Keycloak?

Keycloak es un *servidor de identidad y autenticación* (IAM / Authorization Server) de código
abierto. En lugar de que cada aplicación guarde usuarios y contraseñas, esa responsabilidad se
delega en Keycloak. Tu aplicación deja de preguntar "¿esta contraseña es correcta?" y pasa a
preguntar "¿este token que me trae el usuario es válido?".

Keycloak se encarga de:

* Guardar usuarios, contraseñas (hasheadas con argon2), correos y datos de perfil.
* Verificar credenciales y emitir *tokens* (credenciales temporales firmadas).
* Permitir login social (Google, GitHub, etc.) y SSO (un solo login para varias apps).
* Gestionar roles y permisos a nivel de identidad.

[TIP]
====
La idea central: *Keycloak prueba quién sos (autenticación). Tu aplicación decide qué podés hacer
(autorización).* En este proyecto los roles de negocio viven en la base de datos de la app, y
Keycloak solo aporta la identidad.
====

== El concepto clave: el Realm

Un *realm* (reino) es un *espacio aislado* dentro de Keycloak. Todo lo que vive dentro de un realm
—usuarios, roles, clientes, configuración de tokens— es independiente de los demás realms.

Pensalo como *edificios separados*: un usuario del realm A no existe en el realm B. Cada realm
tiene sus propias llaves de firma, sus propios usuarios y su propia configuración de seguridad.

[cols="1,3"]
|===
|Regla |Significado

|Aislamiento total
|Usuarios, clientes y roles de un realm no se ven desde otro realm.

|Llaves propias
|Cada realm firma sus tokens con su propio par de llaves. Un token de un realm no vale en otro.

|`master` es especial
|Keycloak trae un realm `master` solo para administrar Keycloak. *Nunca* pongas tus apps ahí.
|===

.Buena práctica
[NOTE]
====
Creá *un realm por proyecto/organización*, no uno por aplicación. Varias apps de la misma
organización (web, mobile, API) comparten el mismo realm para tener SSO entre ellas.

En este proyecto el realm se llama `spring-boot-template` (definido en
`keycloak/realm-export.json` y en `keycloak.realm` de `application.properties`).
====

=== Jerarquía: cómo se organiza todo

----
Keycloak (el servidor)
└── Realm: spring-boot-template        ← espacio aislado de este proyecto
    ├── Users                          ← personas: user, admin, ...
    ├── Roles (realm roles)            ← ADMIN, CLIENT_CREATE, CLIENT_GET, ...
    ├── Clients                        ← aplicaciones que hablan con Keycloak
    │   ├── spring-app                 ← la app/front que hace login de usuarios
    │   └── resource-server            ← la API (service account)
    ├── Client Scopes                  ← qué claims/roles entran en el token
    └── Identity Providers
        └── google                     ← broker hacia Google OAuth2 (configurado)
----

== Los actores dentro del Realm

=== Users (usuarios)

Las personas que se autentican. Tienen credenciales, correo y *roles asignados*. En el realm de
ejemplo hay dos:

[cols="1,2"]
|===
|Usuario |Roles

|`user`
|`CLIENT_CREATE`, `CLIENT_GET`, `CLIENT_SEARCH`, `UPDATE_BALANCE`

|`admin`
|`ADMIN`
|===

WARNING: Estos usuarios con contraseñas triviales (`user`/`password`, `admin`/`admin`) son *solo
para desarrollo*. Un realm de producción no debe importar usuarios de prueba.

=== Clients (clientes)

Un *client* representa a una *aplicación* que confía en Keycloak (no a una persona). Hay dos tipos:

[cols="1,1,2"]
|===
|Tipo |Tiene secreto |Ejemplo

|*Confidential* (confidencial)
|Sí (`client-secret`)
|Backends que pueden guardar un secreto de forma segura.

|*Public* (público)
|No
|SPAs/mobile, donde no se puede esconder un secreto.
|===

El realm tiene tres clients:

[cols="1,1,2"]
|===
|Client |Tipo |Para qué sirve

|`spring-app`
|Confidencial
|Login de usuarios con usuario + contraseña (flujo `directAccessGrants`). También tiene
`standardFlow` activado. Usado por `/api/auth/login` y `/api/auth/register`.

|`resource-server`
|Confidencial
|El backend actúa como cliente OAuth2 para el flujo de autorización con Google. Tiene
`serviceAccountsEnabled` (cuenta de máquina con permiso `manage-users` para la Admin API) y
`redirectUri: http://localhost:8081/api/auth/callback`. Se usa también para introspección de tokens.

|`spring-boot-template`
|*Público* (SPA)
|Cliente público con PKCE (`S256`). Pensado para que el frontend pueda iniciar el flujo de
autorización directamente. `redirectUris`: `http://localhost:4200/auth/callback` y
`http://localhost:8081/api/auth/callback`. `webOrigins`: `http://localhost:4200`.
|===

=== Roles

Un *role* es una etiqueta de permiso. Hay dos niveles:

* *Realm roles*: globales al realm (ej. `ADMIN`).
* *Client roles*: específicos de un client.

Los roles del usuario viajan dentro del token (en el claim `realm_access.roles`, configurado por
un *protocol mapper* del scope `roles`).

=== Client Scopes y Protocol Mappers

Un *client scope* agrupa la información que se mete en el token. Un *protocol mapper* es la regla
concreta que dice "tomá este dato del usuario y ponelo en este claim del token". Ejemplo del realm:
el scope `roles` tiene un mapper que copia los roles del realm al claim `realm_access.roles`.

== ¿Qué es un token y por qué hay varios?

Cuando un usuario se autentica, Keycloak emite *tokens*: cadenas firmadas que prueban la identidad
por un tiempo limitado.

[cols="1,1,2"]
|===
|Token |Vida |Para qué

|*Access Token (AT)*
|Corta (5 min aquí)
|Se envía en cada request a la API. Si lo roban, sirve poco tiempo.

|*Refresh Token (RT)*
|Larga
|Sirve para pedir un nuevo AT sin volver a escribir la contraseña.

|*ID Token*
|Corta
|Datos del usuario (OpenID Connect). Lo consume el cliente, no la API.
|===

.Rotación de refresh tokens (configurado en este realm)
[NOTE]
====
`revokeRefreshToken: true` + `refreshTokenMaxReuse: 0` activan la *rotación*: cada vez que usás un
RT para refrescar, Keycloak invalida el anterior y emite uno nuevo. Si alguien roba un RT viejo y lo
reutiliza, Keycloak detecta el reúso y corta la sesión. Es una protección importante.
====

== Cómo se obtiene un token: los "flows"

Un *flow* (flujo / grant type) es el procedimiento por el cual una app consigue un token. Los dos
relevantes acá:

=== 1. Password Grant (Direct Access Grants / ROPC)

La app manda usuario + contraseña directo a Keycloak y recibe los tokens.

----
Usuario → App → Keycloak (/token con username+password) → tokens
----

Es el flujo que usa el endpoint `/api/auth/login` de este proyecto. Es simple, pero está
*desaconsejado en OAuth 2.1* porque la app ve la contraseña. Solo es aceptable para apps de
*primera parte* (tuyas y confiables).

=== 2. Authorization Code Flow (con redirección)

El usuario es *redirigido a la pantalla de Keycloak* (o de Google), escribe sus credenciales ahí, y
vuelve a la app con un código que se canjea por tokens. La app *nunca ve la contraseña*.

----
Usuario → App → (redirige a) Keycloak/Google → login → (redirige de vuelta con code) → App → tokens
----

Es el flujo *recomendado* y el que se necesita para *login social*. En este realm `spring-app`
tiene `standardFlow` activado, pero faltan `redirectUris` y un Identity Provider (Google) para que
funcione de punta a punta.

== Cómo encaja Keycloak en este proyecto

La app soporta tres modos de seguridad (propiedad `app.security.mode`):

[cols="1,2,2"]
|===
|Modo |Usa Keycloak |Cómo valida

|`API_KEY`
|No
|Clave de API guardada en la base de datos de la app. Keycloak no participa.

|`KEYCLOAK_OPAQUE` (default)
|Sí
|En cada request, la API pregunta a Keycloak `/introspect` si el token sigue vivo (revocación en
tiempo real).

|`KEYCLOAK_JWT`
|Sí
|La API valida la firma del token localmente con las llaves públicas del realm (`/certs`), sin
llamar a Keycloak en cada request.
|===

.Opaque vs JWT — la diferencia práctica
[cols="1,2"]
|===
|Aspecto |Resumen

|Opaque (introspección)
|Más seguro (revocación inmediata), pero una llamada de red por request.

|JWT (firma local)
|Más rápido (sin red), pero el token sigue siendo válido hasta que expira aunque se revoque.
|===

== Glosario rápido

[cols="1,3"]
|===
|Término |Significado en una línea

|Realm
|Espacio aislado con sus propios usuarios, clientes, roles y llaves.

|Client
|Una aplicación (no una persona) que confía en Keycloak.

|Service Account
|Un "client" que actúa como máquina, sin usuario humano detrás.

|Token
|Credencial temporal firmada que prueba la identidad.

|Claim
|Un dato dentro del token (ej. `sub`, `email`, `realm_access.roles`).

|Scope / Mapper
|Reglas que deciden qué datos del usuario entran en el token.

|Introspection
|Preguntarle a Keycloak en vivo si un token sigue siendo válido.

|Identity Provider (IdP)
|Un proveedor externo de login, como Google o GitHub.

|Grant / Flow
|El procedimiento para obtener un token (password, authorization code, etc.).
|===

== Advertencia: "You are logged in as a temporary admin user"

Esta advertencia aparece porque el admin configurado en `docker-compose.yaml` vía
`KC_BOOTSTRAP_ADMIN_USERNAME` es un *bootstrap admin* — un usuario temporal diseñado solo para el
primer arranque. En Keycloak 26+ todos los admins creados por esta variable son temporales por
diseño, y Keycloak siempre muestra el aviso.

*En desarrollo se puede ignorar.* No afecta funcionalidad.

Si se quiere eliminar el warning (necesario en producción):

. Loguearse en `http://localhost:8080` con el bootstrap admin.
. Cambiar al realm *master* (arriba a la izquierda).
. Ir a *Users → Add user*, crear un nuevo usuario (ej. `superadmin`).
. En la pestaña *Credentials*, asignarle una contraseña permanente.
. En la pestaña *Role mappings*, asignar el rol `admin` del realm `master`.
. Cerrar sesión y volver a entrar con el nuevo usuario → el warning desaparece.
. Opcional: borrar el bootstrap admin.

[NOTE]
====
Renombrar las vars de `KEYCLOAK_ADMIN` a `KC_BOOTSTRAP_ADMIN_USERNAME` elimina warnings de
deprecación pero *no* elimina el aviso de admin temporal — ambas crean el mismo tipo de usuario.
====

== Probarlo localmente

Levantá Keycloak (importa el realm automáticamente):

[source,bash]
----
docker compose --profile keycloak up
----

* Consola de administración: `http://localhost:8080` (usuario/clave de admin según tu `.env`).
* Dentro de la consola, seleccioná el realm `spring-boot-template` arriba a la izquierda para ver
  sus usuarios, clientes y roles.

Pedir un token a mano (flujo password):

[source,bash]
----
curl -X POST \
  http://localhost:8080/realms/spring-boot-template/protocol/openid-connect/token \
  -d "client_id=spring-app" \
  -d "client_secret=<secret-de-spring-app>" \
  -d "grant_type=password" \
  -d "username=user" \
  -d "password=password"
----

La respuesta trae `access_token` y `refresh_token`. Ese mismo intercambio es el que hace por vos el
endpoint `/api/auth/login`.

== Login con Google: configuración paso a paso

Esta sección documenta los pasos exactos para habilitar el login social con Google en este proyecto.
Son tres etapas: Google Cloud Console → Keycloak Identity Provider → variables de entorno.

=== 1. Google Cloud Console — crear las credenciales OAuth2

. Ir a https://console.cloud.google.com y seleccionar (o crear) un proyecto.
. En el menú izquierdo: *APIs & Services → Credentials*.
. Clic en *Create Credentials → OAuth client ID*.
. En *Application type* seleccionar *Web application*.
. En *Authorized redirect URIs* agregar exactamente:
+
----
http://localhost:8080/realms/spring-boot-template/broker/google/endpoint
----
+
Este es el endpoint del broker de Keycloak. Google redirige aquí luego del login.
. Guardar. Google devuelve un *Client ID* y un *Client Secret* — copiarlos.

[NOTE]
====
Para producción hay que agregar la URL pública de Keycloak en lugar de `localhost:8080`.
La estructura siempre es: `{keycloak-url}/realms/{realm}/broker/google/endpoint`.
====

=== 2. Keycloak — configurar el Identity Provider Google

. Abrir la consola de Keycloak: `http://localhost:8080`.
. Seleccionar el realm *spring-boot-template* (arriba a la izquierda).
. En el menú izquierdo: *Identity Providers → Add provider → Google*.
. Completar los campos:
+
[cols="1,1,2"]
|===
|Campo |Valor |Por qué

|Alias
|`google`
|Debe coincidir con el valor de `kc_idp_hint` que envía el backend.

|Client ID
|El Client ID de Google Cloud Console
|Identifica a Keycloak ante Google.

|Client Secret
|El Client Secret de Google Cloud Console
|Secreto compartido entre Keycloak y Google.

|Prompt
|`select_account`
|Keycloak le pasa este parámetro a Google para que el usuario pueda elegir cuenta.

|Request refresh token
|*On*
|Hace que Google incluya un `refresh_token` en la respuesta al broker.
Necesario para que Keycloak tenga sesión larga con Google.

|Store tokens
|*Off*
|No es necesario guardar los tokens de Google en la BD de Keycloak.
El proyecto solo necesita que Keycloak emita sus propios tokens.

|Trust Email
|*On*
|Evita que Keycloak exija verificar el email de cuentas de Google (ya lo verificó Google).

|Scopes
|(vacío)
|Keycloak usa sus defaults (`openid profile email`). No hace falta sobreescribir.
|===
+
. Guardar.

Keycloak ya sabe cómo hablar con Google. El `kc_idp_hint=google` que envía el backend en la URL
de autorización hace que Keycloak salte su propia pantalla de login y redirija directo a Google.

=== 3. Variables de entorno — lo que necesita el backend

El backend no habla con Google directamente. Habla con Keycloak usando el client `resource-server`.

[IMPORTANT]
====
La configuración de *Keycloak* (redirect URIs válidas, client secret del client, scopes) viene del
`keycloak/realm-export.json`, que Keycloak importa automáticamente al arrancar con Docker Compose.
*No hay nada que configurar a mano en la UI de Keycloak para el client `resource-server`.*

Las variables de entorno que siguen son solo para el *backend Spring Boot* — le dicen qué valores
usar cuando construye las requests hacia Keycloak.
====

[source,bash]
----
# Le dice al backend con qué client autenticarse al intercambiar el código OAuth2
KEYCLOAK_RESOURCE_CLIENT_ID=resource-server
KEYCLOAK_RESOURCE_CLIENT_SECRET=<secret-del-client-resource-server>

# DEBE coincidir con el redirectUri del client en realm-export.json.
# Si no coinciden, Keycloak rechaza el intercambio (redirect_uri mismatch).
KEYCLOAK_OAUTH2_REDIRECT_URI=http://localhost:8081/api/auth/callback

# A dónde redirige el backend al usuario una vez que tiene el access_token
KEYCLOAK_OAUTH2_FRONTEND_CALLBACK_URL=http://localhost:4200/auth/callback
----

El secret de Google va solo dentro de Keycloak (paso 2). El backend no lo toca.

=== 4. Flujo completo end-to-end

Este es el recorrido exacto de una autenticación con Google en este proyecto:

----
[1] Frontend        GET /api/auth/google/initiate
                    ← { authorizationUrl: "https://keycloak/.../auth?...&kc_idp_hint=google&state=XYZ" }

[2] Frontend        Redirige el browser a authorizationUrl

[3] Keycloak        Ve kc_idp_hint=google → omite su pantalla, redirige a Google

[4] Google          Muestra login (con prompt=select_account, el usuario puede elegir cuenta)

[5] Google          Autentica → redirige al broker de Keycloak:
                    GET /realms/spring-boot-template/broker/google/endpoint?code=...

[6] Keycloak        Crea o actualiza el usuario en el realm → redirige al backend:
                    GET /api/auth/callback?code=AUTH_CODE&state=XYZ

[7] Backend         Valida CSRF state (cookie vs. query param)
                    POST /token con grant_type=authorization_code, client_id=resource-server
                    ← { access_token, refresh_token, expires_in }

[8] Backend         Escribe refresh_token en cookie HttpOnly SameSite=Lax
                    Redirige a: http://localhost:4200/auth/callback?access_token=AT&expires_in=300

[9] Frontend        Lee access_token del query param, lo guarda en memoria
                    Limpia la URL con replaceState
----

.¿Por qué SameSite=Lax y no Strict?
[NOTE]
====
El paso [6] es una redirección GET cross-site (Google → Keycloak → localhost:8081). Con
`SameSite=Strict` el browser bloquea el envío de cookies en estas navegaciones de primer nivel
provenientes de otro origen. `SameSite=Lax` permite el envío en navegaciones GET de primer nivel,
que es exactamente el caso aquí. Por eso el cookie del refresh token usa `Lax`.
====

.¿Por qué hay dos URLs de Keycloak (auth-server-url vs public-auth-server-url)?
[NOTE]
====
`keycloak.auth-server-url` se usa para llamadas servidor-a-servidor (token endpoint, introspection,
Admin API) — puede ser una URL de red interna. `keycloak.public-auth-server-url` se usa para
construir URLs que *el browser del usuario* va a seguir — tiene que ser una URL accesible desde
el cliente. En desarrollo ambas son `http://localhost:8080`.
====

=== 5. Probarlo localmente

Con Docker Compose corriendo y el Identity Provider de Google configurado en Keycloak:

[source,bash]
----
# 1. Pedir la URL de autorización
curl http://localhost:8081/api/auth/google/initiate

# La respuesta tiene authorizationUrl. Abrirla en el browser → Google login.
# Keycloak redirige a /api/auth/callback → el backend escribe la cookie y redirige al frontend.
----

Para verificar que el Identity Provider está activo en Keycloak:

[source,bash]
----
# Listar los identity providers del realm
curl http://localhost:8080/realms/spring-boot-template/.well-known/openid-configuration \
  | python3 -m json.tool | grep authorization_endpoint
----
